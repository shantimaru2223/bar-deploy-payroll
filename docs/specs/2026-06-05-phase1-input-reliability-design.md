- **対象アプリ**: Bar Deploy 給与明細アプリ
- **作成日**: 2026-06-05
- **作成経緯**: brainstorming スキルでオーナーと合意した設計。実装計画は別途 writing-plans スキルで作成する。
- **位置づけ**: 「ブラッシュアップ」全体のうち Phase 1（土台固め）。Phase 2＝見た目の刷新、Phase 3＝新機能は本書の対象外。

---

## 1. 目的・成功基準

### 目的
スタッフ情報を画面から安全に「編集」でき、かつ不正な入力を画面・サーバの両方で防ぐことで、誤入力・計算ミスを減らし、運用の信頼性を上げる。

### 成功基準（Definition of Done）
1. スタッフ情報を画面から編集でき、「削除→再登録」が不要になる。
2. 不正な入力（負の金額・非数値・不正な時刻形式・範囲外の日数）が **画面とサーバの両方** で弾かれる。
3. 「終了 ≦ 開始」は翌日終了として計算しつつ、画面に「翌日終了として計算（◯時間◯分）」と明示され、誤入力に気づける。
4. 給与計算が純粋関数 `calcPayroll()` に切り出され、`node:test` のユニットテストが全て通る（挙動はリファクタ前と同一）。
5. `docs/design.md`・`docs/requirements.md` が現状仕様（3給与タイプ・ドリンクバック・片道交通費・源泉の課税範囲・`work_minutes`）に一致する。

## 4. 機能仕様

### 4.1 スタッフ編集
- **API**: `PUT /api/staff/:id`
  - 更新項目: `name`, `pay_type`, `hourly_rate`, `monthly_salary`, `daily_rate`, `drink_back_rate`, `transport_fee`
  - 存在しない `id` は `404 { error }`
  - バリデーション通過後に `UPDATE`、成功で `{ success: true }`
- **画面**: スタッフ一覧の各行に「編集」ボタンを追加。クリックで既存の登録フォームを **編集モード** として再利用し、現在値を読み込む。保存で `PUT` → 一覧を再描画。
- **合意事項（留意）**: 明細は「現在のスタッフ設定」で都度計算されるため、設定変更は **過去月の明細にも反映される**。1〜5名のオーナー運用のため許容とする。

```
[一覧]  山田太郎  時給1,200円  [編集] [削除]
                     │ クリック
                     ▼
[フォーム:編集モード] 既存値が入った状態 → 修正 → [保存]
                     │ PUT /api/staff/:id
                     ▼
            一覧を再描画（最新値）
```

### 4.2 入力バリデーション（クライアント＝即時／サーバ＝安全網）

| 対象 | ルール | 違反時 |
|---|---|---|
| スタッフ名 | 必須（trim後1文字以上） | 赤字で指摘 |
| 各金額（時給/月給/日給/ドリンク単価/交通費・片道） | 0以上の整数のみ | 弾く＋指摘 |
| 給与タイプ | `hourly` / `daily` / `monthly` のいずれか | 弾く |
| 勤務日 | `YYYY-MM-DD` 形式・実在日 | 弾く |
| 開始/終了時刻 | `HH:MM`（00:00〜23:59） | 弾く |
| 終了 ≦ 開始 | 翌日終了として +24h 計算し「翌日終了として計算（◯時間◯分）」を表示 | 警告表示（ブロックしない） |
| 出勤日数（月次） | 0〜31 の整数 | 弾く |
| ドリンク杯数（月次） | 0以上の整数 | 弾く |

- **サーバ**: 各APIで検証し、違反時は `400 { error }` を返す。
- **クライアント**: 送信前に検証し、各入力欄の下に赤字インラインで表示。妥当になるまで送信不可。
- **重複の最小化**: 検証ルールを `src/validation.js` に集約し、サーバは `require` で利用。素のHTML/JSのためクライアントは完全共有はせず、主要ルールを同等に再現する（**サーバ側を正**とする）。
- **給与額の補足**: 金額は「0以上の整数」を必須とする。選んだ給与タイプに対応する単価（例：時給スタッフの `hourly_rate`）が 0 の場合は **警告のみ**（ブロックしない）。

### 4.3 フォームUX調整（軽め）
- 必須項目に `*` を表示。エラーは入力欄の直下に赤字インライン。
- 選択中の給与タイプに応じて不要な入力欄を非表示（例：時給選択時は月給・日給欄を隠す）。
- 送信中はボタンを無効化（二度押し防止）。成功時にメッセージ表示。
- スマホ入力強化：`<input type="time">` / `<input type="date">` / `inputmode="numeric"`。
- ※ ここは「分かりやすさ」優先の軽い調整。本格的な見た目刷新は Phase 2 に回す。

## 5. テスト容易性のためのリファクタ
- 現状 `src/app.js` の `GET /api/payroll/:staffId/:yearMonth` 内にある給与計算ロジックを、純粋関数 `calcPayroll(staff, attendances, monthly)` として切り出す（`src/payroll.js` 新規）。
  - 戻り値: `workDays, totalMinutes, basePay, drinkCount, drinkBack, transportFare, transportFee, taxableBase, withholdingTax, grossPay, netPay`
- ルートは関数を呼ぶだけにする。**リファクタ前後で出力は完全に同一**に保つ。
- 利点: DB・Express を起動せずに計算を検証可能（`better-sqlite3` のビルド問題にも左右されない）。

## 6. テスト計画
- 追加依存なしの `node:test`（`node --test`）。
- `calcPayroll`: 時給／日給／月給 × ドリンクバック × 交通費 × 源泉、境界値（出勤0日、金額0、日またぎシフト）。
- `validation`: 各ルールの正常系・異常系・境界値。
- 手動／preview: 編集フロー、バリデーションのインライン表示、タイプ別の入力欄出し分け。

## 8. リスク・留意点
- `better-sqlite3` はローカル Node 26 でビルド不可（過去ログより）。→ 計算・検証を **DB非依存の純粋関数** にすることで、Node バージョンに左右されずテスト可能にする。
- Render は master から自動デプロイ。→ 本作業は **feature ブランチ（`phase1-input-reliability`）** で行い、完了・確認後に master へ反映する。
- スタッフ設定の変更が過去明細に反映される件は、仕様として本書に明記済み（4.1）。
